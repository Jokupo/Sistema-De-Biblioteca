💡 DECISÕES DE DESIGN
═════════════════════

1. Menu Hierárquico (Principal + 4 Secundários)
   → Razão: Agrupa funcionalidades relacionadas
   → Benefício: UX melhor, navegação clara

2. Separação de Livros Disponíveis vs Emprestados
   → Razão: Informação importante para usuário
   → Benefício: Consultas rápidas, clareza

3. Busca Parcial por Título
   → Razão: Usuário não lembra título exato
   → Benefício: Encontra livro mais facilmente

4. Validação de Duplicação
   → Razão: Garantir integridade de dados
   → Benefício: Sem dados duplicados

5. Confirmação antes de Salvar
   → Razão: Evitar erros em registro da API
   → Benefício: Usuário valida antes de confirmar

6. Feedback Específico com Emojis
   → Razão: Melhorar compreensão
   → Benefício: Interface mais amigável

🎓 PADRÕES E PRINCÍPIOS APLICADOS
══════════════════════════════════

✅ SOLID Principles:
   • S - Single Responsibility: Cada método com uma função clara
   • O - Open/Closed: Pronto para extensão futura
   • D - Dependency Inversion: Biblioteca gerencia dados

✅ DRY (Don't Repeat Yourself):
   • buscarUsuarioPorEmail() reutilizado por procurarUsuario()
   • Constantes evitam strings hardcoded
   • Métodos de validação centralizados

✅ KISS (Keep It Simple, Stupid):
   • Métodos com responsabilidade clara
   • Nomes descritivos
   • Lógica fácil de seguir

✅ Clean Code:
   • Formatação consistente
   • Variáveis com nomes significativos
   • Comentários onde necessário
